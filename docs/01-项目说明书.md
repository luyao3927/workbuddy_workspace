## 一、项目概述

### 1.1 项目背景

随着餐饮行业数字化进程加速，越来越多中小型私房菜馆希望通过低成本、轻量化的方式实现顾客自助点餐。传统的纸质菜单存在更新不便、菜品展示单一、顾客点餐体验差等痛点。

本项目以「私房菜馆」为业务原型，基于 **uniapp + Vue 3 + Pinia** 技术栈开发，目标平台为**微信小程序**，提供一套可直接商用交付的电子菜谱自助点餐模板。项目聚焦核心点餐链路，采用**手帐/collage 风格 + 喜茶清新年轻化**视觉设计，打造具有品牌差异化的用户体验。

### 1.2 项目目标

- **快速交付**：模板化设计，商家可快速配置上线
- **体验优先**：手帐风格视觉设计，区别于传统点餐小程序的同质化 UI
- **轻量运行**：聚焦 P0 核心功能，不引入不必要的复杂度
- **易于对接**：预留标准化 API 接口，可快速对接 WorkBuddy 数据平台

### 1.3 产品定位

| 维度       | 说明                                               |
| ---------- | -------------------------------------------------- |
| 目标用户   | 25-40 岁年轻女性为主，注重用餐体验和视觉审美       |
| 适用门店   | 私房菜馆、网红餐厅、精品小馆                       |
| 品牌调性   | 清新、手作、年轻化、有温度                         |
| 核心价值   | 纸质菜单电子化 + 顾客自助点餐 + 品牌视觉差异化     |

## 二、适用场景与目标用户

### 2.1 适用场景

1. **到店扫码点餐**：顾客入座后扫描桌上二维码，进入电子菜谱自主浏览和点餐
2. **菜品浏览与推荐**：通过分类导航、推荐标签快速发现心仪菜品
3. **规格定制**：支持菜品规格选择（如口味、份量），满足个性化需求
4. **购物车管理**：实时查看已选菜品、修改数量、合计价格
5. **订单提交**：确认订单信息后一键提交至商家

### 2.2 目标用户画像

| 角色       | 特征                                               | 核心需求                             |
| ---------- | -------------------------------------------------- | ------------------------------------ |
| 用餐顾客   | 25-40 岁女性为主，习惯移动端操作，注重视觉体验     | 快速浏览菜品、便捷下单、美观的界面   |
| 餐厅老板   | 中小型私房菜馆经营者，无专业技术团队               | 低成本上线、易维护、支持日常运营     |
| 运营人员   | 负责菜品更新、菜单维护                             | 通过后台管理系统更新菜品信息         |

## 四、技术架构说明

### 4.1 技术栈

| 层级         | 技术选型                             | 说明                     |
| ------------ | ------------------------------------ | ------------------------ |
| 前端框架     | Vue 3 (Composition API)              | 最新版本，使用组合式 API |
| 跨端框架     | uniapp                               | 一套代码多端运行         |
| 状态管理     | Pinia                                | Vue 官方推荐状态管理库   |
| 构建工具     | Vite (uniapp 内置)                   | 快速开发构建             |
| 样式方案     | SCSS + CSS Variables                 | 支持暗黑模式切换         |
| HTTP 请求    | uni.request (封装)                   | 统一请求拦截与错误处理   |
| 数据模拟     | Mock 数据 (可切换)                   | 开发阶段独立运行         |
| 目标平台     | 微信小程序                           | 仅手机端                 |
| 版本管理     | Git                                  | 规范化提交               |

### 4.2 架构分层

```
┌─────────────────────────────────────────────┐
│                  Pages (页面层)              │
│   index / dish-detail / order /             │
│   order-success                             │
├─────────────────────────────────────────────┤
│               Components (组件层)            │
│   common / dish / category / cart /         │
│   order / collage                           │
├─────────────────────────────────────────────┤
│               Composables (逻辑层)           │
│   useTheme / useCart / useDish /            │
│   useOrder / usePageState                   │
├─────────────────────────────────────────────┤
│               Stores (状态层)                │
│   useAppStore / useCategoryStore /          │
│   useDishStore / useCartStore /             │
│   useOrderStore                             │
├─────────────────────────────────────────────┤
│               API (接口层)                   │
│   dish / category / order / mock-data       │
├─────────────────────────────────────────────┤
│               Utils (工具层)                 │
│   request / storage / format /              │
│   validator / theme                         │
└─────────────────────────────────────────────┘
```

### 4.3 数据流说明

```
用户操作 → 页面/组件
              │
              ▼
        Composable 逻辑层
              │
              ▼
         Pinia Store (状态管理)
              │
              ▼
         API 层 (请求封装)
              │
              ▼
         Mock 数据 / WorkBuddy 接口
              │
              ▼
         响应数据 → Store 更新 → 页面响应式刷新
```

#### 数据流关键路径

1. **菜品浏览流程**：首页加载 → `useDishStore.fetchDishes()` → API 请求 → Store 存储 → 组件渲染
2. **加入购物车流程**：选择规格 → `useCartStore.addToCart()` → 本地状态更新 → 购物车浮窗刷新
3. **提交订单流程**：确认订单 → `useOrderStore.submitOrder()` → API 请求 → 跳转成功页
4. **主题切换流程**：切换按钮 → `useTheme.toggleTheme()` → CSS 变量切换 → 全局样式刷新

```
┌─────────────┐
│  useAppStore │  应用全局状态（主题、门店信息、加载状态）
└──────┬──────┘
       │
┌──────▼──────────┐

└──────┬──────────┘
       │
┌──────▼──────┐
│ useDishStore │  菜品数据（列表、详情、推荐）
└──────┬──────┘
       │
┌──────▼──────┐
│ useCartStore │  购物车数据（商品列表、总价、数量）
└──────┬──────┘
       │
┌──────▼───────┐
│useOrderStore │  订单数据（当前订单、历史订单）
└──────────────┘
```

### 7.2 数据流转关键节点

| 节点             | 触发时机             | 数据变化                             |
| ---------------- | -------------------- | ------------------------------------ |
| 首页加载         | onLoad               | 请求分类列表 + 推荐菜品列表           |

| 加入购物车       | 规格选择后确认       | 购物车列表新增/更新商品              |

| 删除购物车商品   | 点击删除按钮         | 购物车商品移除，总价重新计算          |
| 提交订单         | 点击提交订单按钮     | 购物车数据转为订单数据，购物车清空    |
| 主题切换         | 点击主题切换按钮     | CSS 变量集合切换，全局样式变更        |

### 9.3 切换 Mock 数据模式

编辑 `config/api.config.js`：

```javascript
export default {
  USE_MOCK: true,           // true: 使用本地 Mock 数据 / false: 使用远程接口
  BASE_URL: 'https://api.example.com', // API 基础地址
  TIMEOUT: 10000,           // 请求超时时间（ms）
}
```

### 9.4 菜品数据配置

Mock 数据定义在 `api/mock-data.js`，包含：
- 分类数据（categories）
- 菜品数据（dishes，含图片、价格、规格、标签等）
- 推荐菜品数据

对接 WorkBuddy 真实数据后，Mock 数据将被接口数据替代。
